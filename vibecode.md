## 这篇文章到底在讲什么

一句话：**怎么让 AI 在一个真实的、跨多个代码仓库的大项目里，从头到尾稳定地跑完一个需求**。

背景是这样的：现在模型越来越强了，但你真让 AI 独立完成一个完整的产品需求——从接到需求单开始，到设计方案、改代码、跑测试、通过代码审查、最后提交合并——你会发现**模型够强了，但工程没跟上**。AI 跑不完，因为这条链路上太多事情不是"写代码"本身，而是协作、流程、信任、收口。

他们的 TAB 平台是个典型的大工程：30+ 个微服务、10+ 个前端微应用，全部放在一个大仓库（monorepo）里统一管理。在这种规模下让 AI 干活，会遇到一堆具体问题。

## 第一章：他们的工程长什么样

TAB 是腾讯内部的 A/B 实验平台——产品同事发起一个实验、配人群、配指标、灰度推全量，背后是一整套服务体系。

技术上几个关键点：
- **大仓 + 多子仓库**：一个主仓库下面挂着好多个子模块，实验编排服务、人群服务、指标服务、网关、前端应用各占一块
- **多语言**：主要 Go 和 TypeScript，还有一些 Python 工具链和各平台 SDK
- **四层架构**：胶水层 → 接口层 → 服务层 → 领域层
- **验证手段**：一键拉起完整沙箱环境（Docker Compose），能跑真实的 HTTP 接口测试

**真正让他们头疼的不是"AI 写代码不准"**，而是这四件事：

1. **需求单不可信**：产品给你一句话"加个白名单删除"，但删除后会不会影响下游？灰度怎么处理？接口幂等吗？没人说清
2. **方案对不上代码**：方案文档写得很漂亮，但具体改哪几个服务、加几个字段、怎么跟现有架构对齐——看不出来
3. **改完没人验证**："我改完了"和"我跑过沙箱测试且覆盖率达标"是两回事。前者是 AI 的口头汇报，后者是机器验证的事实
4. **交付环节碎一地**：写完代码还得去改需求单状态、上传方案文档、提 MR、贴度量数据——动作都不难，但全靠人记，必然漏

注意：**这四件事没有一件是模型问题，全是协作、流程、信任问题**。

所以他们要搭的这套 Harness（你可以理解为"AI 工作的脚手架+约束体系"），**主要不是给 AI 用的，是给团队用的**。目标不是让 AI 看起来更聪明，而是让 AI 在这个工程里**稳定、规范、可审计地把事情做对**。

## 第二章：Harness 由哪几块东西组成

他们把整套体系拆成了 6 层资产：Rule（规则）、Skill（操作手册）、Sub Agent（专职角色）、Workflow（流程定义）、Scripts（门禁脚本）、MCP（外部系统接口）。逐个讲：

### Rule（规则）：他们最后反而没怎么写

这个反直觉。Rule 就是写给 AI 看的自然语言约束，比如"数据访问层必须用 GORM，不许直接拼 SQL"。

问题是：**自然语言天然能被 AI 找理由绕过去**。AI 可以说"这条规则在这个场景不适用""我做了等价的事""这是历史问题"——每一句都可能是真的，但你没法每次都去翻 git 记录验证。

所以他们的做法是：**能判定的规则全部下沉成脚本或 lint 检查器**，独立的 Rule 反而最少。比如：
- "必须用统一错误码包" → 一个 lint 检查器搞定
- "覆盖率 ≥70%" → 脚本强制检查
- "写操作必须走事务" → 代码审查阶段静态扫描

**核心原则：能用机器判定的，就别留在自然语言规则里**。

### Skill（操作手册）：管理注意力，不是增强能力

他们只留了 11 个 Skill，分四组：代码层规范（4个）、测试规范（2个）、需求理解工具（3个）、代码审查能力（2个）。

Skill 解决的核心问题是：**把"这件事具体怎么做"从主 Agent 的大脑里拆出去**。

举个例子：数据访问层有一套很硬的规矩——必须用 GORM、必须用 mock 写单测、增量覆盖率 ≥80%、错误必须用统一错误码包。如果把这些全塞进开发 Agent 的主 prompt，主 prompt 越来越长，真正该关心的需求上下文反而被淹没。

所以独立成一份"数据访问层规范手册"，Agent 做到数据访问任务时按需加载。**Skill 的本质是 AI 的注意力管理——让主 Agent 只关注当下该关注的事**。

### Sub Agent（专职角色）：只留了 4 个

最终只有 4 个 Agent：**需求 Agent、方案 Agent、开发 Agent、代码审查 Agent**，外加 1 个总控 Agent 做调度。

| 角色 | 干什么 | 不许干什么 |
|------|--------|-----------|
| 需求 Agent | 把需求单变成结构化的需求理解文档 | 不许写技术方案 |
| 方案 Agent | 把需求变成可执行的技术方案 | 不许改需求理解文档 |
| 开发 Agent | 按方案写代码 + 测试 | 不许超出方案范围 |
| 代码审查 Agent | 从 4 个维度审查 | 不许改代码，只能提阻塞项 |

为什么只有 4 个？因为像"跑门禁""跑测试""提 MR"这些事，看上去也像"角色"，但其实是**纯确定性工作**——让 AI 来解释执行只会浪费 token 和增加不确定性。**能写成 bash 脚本的就别让 Agent 跑**。

### Workflow（流程）：13 个阶段的接力赛

整个流程 13 棒：初始化 → 需求分析 → 需求确认 → [Bug 复现] → 技术方案 → [方案确认] → 方案评审 → 分支准备 → 开发 → 集成测试 → 代码审查 → 验收 → 交付收尾。

方括号里的是条件触发——bugfix 类需求才走 Bug 复现，涉及改协议或跨多服务时才弹方案确认。

每棒都明确定义了：当前阶段要交付什么、下一棒接手前必须看到什么、什么情况打回、打回到哪一棒。

### Scripts（门禁脚本）：最硬的那道墙

7 道门禁脚本，后面第六章详细讲。一句话概括：**把"完成"这个词的定义权从 AI 手里拿回来**。

### MCP（外部系统接口）：通往交付闭环的最后一程

接了 5 个外部系统的 MCP：TAPD（需求管理）、iWiki（文档）、工蜂（代码托管+MR）、Knot（设计规范）、企微（通知）。让 AI 能在受控范围内操作这些外部系统。

## 第三章：为什么 13 个阶段一个都不能少

一开始他们也想合并——初始化和需求分析合并？开发和集成测试合并？一个 Agent 顺手就做了嘛。

**每次"合"都被现实打回来**。每个阶段的存在都对应着一类历史教训：

- **初始化**单独存在，是因为它有一组只在最开头跑一次的操作（改需求单状态、打标签），如果和需求分析混合，需求分析被打回时这些状态会被反复改
- **集成测试**单独存在，是因为如果留到验收才发现接口对不上，代码审查已经审过了，全部白审
- **交付收尾**单独存在，是因为"提 MR、改状态、上传文档"这些动作散落在开发末尾，必然漏

### 最值得讲的故事：集成测试从"后置"挪到"前置"

最早没有独立的集成测试阶段，接口测试放在验收阶段。结果出现了一个让人沮丧的循环：

> 开发写完 → 代码审查 Agent 仔细审了 200 行 diff → 审查通过
> → 验收跑接口测试 → 失败（路径写错/参数对不上）
> → 打回开发 → 改 → 重新过代码审查 → ...

代码审查是最贵的环节（每次都要把整个 diff 重看一遍），但接口测试失败其实是个**几秒钟就能验证的事**——却让最贵的环节背了最便宜的锅。

**改法**：接口测试前置到代码审查之前，作为硬门禁。不通过不准进代码审查。

这一改，代码审查的平均打回次数从 1.8 次降到 0.4 次。

更深层的意义是：**"开发完成"的定义变了**——以前是"代码+单测"，现在是"代码+单测+接口用例+接口测试代码"。**让产生问题的人，同时产生验证手段**。

### 反方向的故事：Bug 复现从"必走"变"可选"

Bug 复现最早是所有 bugfix 必走的。跑了几个需求发现：很多 bug 需求单里写得很清楚（请求、期望、实际都有），不需要复现，强制复现等于在沙箱里多花好几分钟跑一遍已经知道答案的事。

**教训：流程要给"明显已经清楚的事"留一条快车道**。

### 阶段切分的判断标准

两条：
1. **它是否有独立的失败模式？**（集成测试失败 ≠ 代码审查失败 ≠ 开发失败）
2. **它失败时回退的位置是否不同？**（集成测试失败回开发、代码审查失败可能回方案）

两个都是"是" → 必须独立。否则就合。

## 第四章：4 个 Agent 是被问题一步步逼出来的

### 第一阶段：一个 Agent 干到底

最早试过"一个 Agent 从头干到尾"。问题很快暴露：
- **上下文混杂**：分不清"需求单写的"和"自己推断的"
- **缺乏制衡**：自己写的方案自己审，几乎不会否决
- **倾向往前推进**：会主动找理由说"问题不大，继续吧"——**这个最致命，它不会主动停下来**
- **token 爆炸**：从头到尾一路，1M 上下文都不够

### 第二阶段：拆出需求和方案 Agent

拆完立刻见效：方案 Agent 会主动指出需求理解文档里哪些边界没说清。

但出了新问题：**方案 Agent 经常觉得需求 Agent 写得不严谨，顺手把验收标准改了**。结果代码审查时不知道哪句标准是产品要的、哪句是方案 Agent 自己加的。

于是定了**最重的一条纪律**：**下游 Agent 不许直接修改上游产物。需要改只能提阻塞项，由总控打回上一棒**。这条纪律解决的不是技术问题，是**责任边界问题**。

### 第三阶段：拆出代码审查 Agent

**写代码的不能自己审自己的代码**——这条几乎是所有多 Agent 设计的硬规矩。

而且代码审查不只是看有没有 bug，它承担的是**整条链的最终收口**，从四个维度审：
1. **方案一致性**：实现是不是真的按设计文档走？有没有偷偷加东西？
2. **验收标准覆盖**：每条验收标准是不是都有对应的代码和测试？
3. **质量基线**：覆盖率达标吗？错误处理符合规范吗？
4. **前端增量**：涉及前端时的专项检查

其中"方案一致性"专门防一类顽固现象：**AI 解释性执行**——AI 实现到一半觉得"设计这里写得不好，我顺手改一下"，从它角度是"优化"，从工程角度是"擅自改设计"。

### 第四阶段：开发 Agent 不只写代码

集成测试前置后，开发 Agent 的职责从"写代码+单测"扩展成了"写代码+单测+接口用例文档+接口测试代码"。

背后的设计哲学：**每个角色对自己产物的"可验证性"负责**。你写了代码，你也得给出验证它的手段。

### 为什么不继续拆

考虑过把开发 Agent 拆成"数据访问层/服务层/处理层"三个并行 Agent。最后没拆，因为：
- 四层架构有强依赖，并行收益不大，协调成本极高
- **Skill 已经解决了分层规范问题**——跑到哪层就加载哪层的 Skill

**能用 Skill 解决的事，就不要再开一个 Agent**。Agent 带状态、带 token 成本；Skill 只是无状态的文档加载。

## 第五章：人工关卡——半自动才是当前最优解

### 为什么不能让 AI 全自动

试过让一个 Reviewer Agent 去判断"方案能不能进开发"。结论：**不行**。

不是模型不够强，是**模型对自己生成的东西天然没有"否决欲望"**。它倾向于推进，不倾向于停下。上游 Agent 写的方案，下游 Agent 99% 的情况会觉得"看上去挺好"。**这个偏置是模型层面的，prompt 改不了**。

所以关键节点必须有人。

### 5 个人工关卡

| 关卡 | 花多久 | 频率 |
|------|--------|------|
| 需求确认 | 10~30 秒 | 每个需求 1 次 |
| 方案确认 | 30~60 秒 | 约 40% 需求触发 |
| 方案评审打回 | 1~2 分钟 | <10% |
| 前端冒烟选择 | 5 秒 | 涉及前端时 |
| 熔断暂停 | 越短越好 | 连续打回 ≥3 次时 |

核心原则：**不能让用户每分钟点一次以上**。体感不对，团队会主动绕过工具退回手动模式。

### 弹窗审批密度的坑

拆完 Agent 后发现：每次"确认继续"，子 Agent 写文件都要弹"确定要写吗？"。一次需求分析弹 5 次，开发阶段弹几十次。体感从"半自动"退化成"全人工"。

原因：调起子 Agent 时没声明权限模式，IDE 默认走最严格的写文件审批。改了一行配置就好了。

教训：**任何让用户每分钟点一次以上的交互，都是设计 bug**。

## 第六章：7 道门禁脚本——最精华的部分

### 为什么脚本会越来越重要

**Rule 越多，AI 的解释空间就越大**。同一条规则，AI 总能找到看上去合理的理由绕过去：
- "这个测试不通过是历史问题"
- "这条规则这次场景特殊"
- "我做了等价验证"

每一句都可能是真的，但你不可能每次都去验证。所以**所有能判定的约束都要下沉成脚本**。"做完"不再是"AI 觉得做完了"，而是"脚本判定通过了"。

### 7 道门禁一览

| 阶段 | 门禁 | 硬/软 | 干什么 |
|------|------|-------|--------|
| 开发完成 | 覆盖率检查 | 软 | 全量 ≥40% 且增量 ≥85% |
| 开发完成 | 制品一致性 | 软 | 任务清单是否全部勾上 |
| 开发完成 | 静态检查+安全扫描 | **硬** | lint 12 个检查器 + 硬编码密码 + SQL 拼接 |
| 集成测试 | 沙箱部署 | **硬** | 拉起完整环境 + 健康检查 |
| 集成测试 | 接口测试 | **硬** | 真实 HTTP 跑接口 |
| 集成测试 | 前端冒烟 | 软 | Playwright 跑路由级渲染 |
| 验收 | 日志验收 | 软 | 关键日志存在 + 敏感信息脱敏 |

**硬门禁只有 3 道，有意为之**。判断标准：不是质量红线的就别阻断流程。流程被频繁打断会让团队怀疑工程化的价值。

### 反作弊设计：基线对比（最精彩的部分）

AI 对失败有顽强的解释能力："这是历史问题""之前就有"。

他们的做法：**开发前先跑一次门禁，冻结成基线快照。开发后再跑一次，只算"新增的失败"**。

> 开发前失败集合 = A
> 开发后失败集合 = B
> 新增失败 = B - A

只有新增的失败才阻断流程。**"历史问题"这把万能钥匙就被拿掉了**。

这个机制的价值不在技术上，**在于剥夺 AI 的解释权**。

### 软门禁也要留"伤疤"

软门禁失败不阻断流程，但会：
- 在状态文件里写警告
- 在代码审查结论里被显式提到
- 在最终度量报告里沉淀

**你不能阻止 AI 偷懒，但你可以让偷懒变得很醒目**。

## 第七章：Team Mode 卡死事件——最值得读的一章

### 症状

人工关卡必现卡死。用户输入"确认继续"后，主 Agent 完全没反应。

### 修了三轮都没修好

- 第一轮：怀疑标记文件残留 → 加清理脚本 → 没用
- 第二轮：怀疑超时设置 → 改长等待 → 没用
- 第三轮：怀疑按钮值匹配 → 改纯文本输入 → 还是没用

每轮都局部止血了某些症状，但根因没动。

### 真相

打开 IDE 的 Team 状态目录看消息收件箱，发现：**Team 还活着，主 Agent 的本轮对话已经结束，但用户输入的"确认继续"安安静静地躺在收件箱里，没有任何 Agent 在消费**。

根因：他们的"启动子 Agent 并等待"机制让主 Agent 的对话跨越了整个子 Agent 执行时间。主 Agent 抛出确认提示后本轮对话自然结束了，但后续用户输入被路由到主 Agent 的收件箱——**可主 Agent 已经下班了**。

> 消息有目的地，但目的地下班了。

### 真正的解法：删掉这个机制

他们盯着架构图看了很久，发现：**4 个 Agent 调用点全部是"串行单个"模式，从来没用过 Team Mode 的任何独有能力**（并行、互发消息、长驻留）。Team Mode 带来的复杂度，**只是为了解决它自己引入的问题**。

经典的**"复杂度自给自足"陷阱**：当一个机制的存在主要是为了解决它自己引入的副作用时，它有 80% 的概率应该被删掉。

最终：彻底删除 Team Mode，改用项目级子 Agent + 同步阻塞调用。删完之后：
- 卡死问题彻底消失
- 辅助脚本从 3 个变成 0 个
- 主 Agent 代码少了 200 多行

### 留下的方法论

1. **永远警惕"自给自足的复杂度"**
2. **先 grep 自己的代码再修 bug**——如果更早确认"一次都没用过 Team Mode 的并行能力"，就不会在时序问题上浪费三轮迭代
3. **YAGNI 原则**——保留可选路径意味着维护两套协议，不如先删掉，未来真需要时再加

## 第八章：让 AI 带上"整个项目的记性"

### 问题：AI 重复造轮子

跑了段时间发现：同一个工具方法，公共包里已经有了，开发 Agent 又写了一遍；同一个错误码已经定义了，又起了一个新的。

不是模型不行，**是它没有"项目记性"**。每次只看当前任务的上下文。

### 两层索引

**第一层：仓库代码导航地图**（~26KB）
由开发 Agent 自己维护，写的不是文件列表，而是"这座工程长什么样"：某类功能落在哪些子仓库、某类服务怎么注册、改某个子仓库可能牵动哪些下游。需求 Agent 启动时先读这份地图，避免 80% 的"在错的子仓库里改"。

**第二层：任务看板**
跨需求的任务总览，记录当前在跑哪些需求、每个到哪个阶段、已完成需求的产物路径。需求 Agent 启动时先扫一眼，避免新需求把旧设计冲掉。

### 为什么不靠 AI 工具的 Memory

仓库里的文档是**团队共识，能审计、能交接**；Memory 是**个人偏好，看不见、对不齐**。4 个人同时跑 Harness，如果项目知识只存在某人的 Memory 里，4 个人的 Memory 会互相矛盾。

纪律：**团队要对齐的东西必须落到仓库，Memory 只放纯个人偏好**。

## 第九章：MCP 是交付闭环的关键

### 没有 MCP 之前

只有"开发闭环"（从需求到代码+测试+CR）。但真实交付还要：改需求单状态、关联分支、上传文档到 iWiki、提 MR、通知群——全靠人做，半小时就过去了。

### MCP 的设计原则

三句话：
1. **读多写少**：能读的尽量多接，写的方法严格挑选
2. **写操作只在初始化和交付收尾触发**：其他阶段一概只读，杜绝"AI 半路改坏外部系统"
3. **调用全部留痕**：每次写操作都记录时间、改了什么字段

### 接 MCP 的几条原则

- **先做开发闭环，再接 MCP**：开发闭环不稳，接了只会让 bug 扩散到外部系统
- **写操作必须幂等**：重复调用不会产生重复数据
- **MCP 失败只警告不阻断**：不允许"网络抖一下，主流程挂掉"
- **MCP 不是 Harness 的主体**：它只是把开发闭环扩展成交付闭环的最后一段路

## 第十章：总结

整套 Harness 压成四块拼图：

| 拼图 | 管什么 | 缺了会怎样 |
|------|--------|-----------|
| 约束与流程 | AI 按什么顺序、什么边界做事 | AI 冲掉接口规约、擅自改设计 |
| 反馈 | 做完后系统能不能判定结果 | "做完了"全靠口头，测试一跑就挂 |
| 知识库 | 项目级记忆 | AI 重复造轮子 |
| 进化 | 演进日志 | 改 prompt 不知道上次为什么这么定 |

### 跑了 50+ 个需求后的数字

- 端到端耗时：30~75 分钟（不含人工等待）
- 人工介入：2~3 次
- 集成测试一次通过率：~70%
- 代码审查阻塞率：~25%
- 开发阶段平均打回：0.4 次

### 给后来人的忠告

**永远不要做的事**：
- 第一天就拆 7 个 Agent
- 让 AI 跑确定性脚本（提交、推送这些全部用 bash）
- 让下游 Agent 改上游产物
- 所有失败都设成硬门禁
- 没有基线对比就信 AI 说"这是历史问题"
- 把团队规约存进 Memory

## 我的观察

这篇文章密度很高，核心讲的不是"怎么用 AI"，而是**怎么管 AI**。几个最有价值的洞察：

1. **"完成"的定义权必须从 AI 手里拿回来** — 靠脚本判定，不靠 AI 汇报
2. **下游不能改上游产物** — 这不是技术约束，是责任边界
3. **剥夺 AI 的解释权** — 基线对比让"历史问题"不再是万能借口
4. **警惕自给自足的复杂度** — 一个机制如果主要在修自己引入的 bug，删掉它
5. **半自动比全自动更难但更值得做** — 当前模型能力下，关键节点必须有人

他们的核心判断标准一句话就能概括：**能用脚本判定的就别用规则描述，能用 Skill 解决的就别开 Agent，能确定性执行的就别让 AI 解释执行**。
